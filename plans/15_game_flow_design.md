## 1. 타겟 플레이어

- **주 타겟**: 코딩을 즐기거나, 해킹/보안 분위기에 흥미를 느끼는 플레이어
- **보조 타겟**: CLI와 스크립팅에 익숙하지 않지만, 시스템을 배우고 자동화하는 재미를 좋아하는 플레이어
- **비대상**: 모든 상호작용이 버튼과 GUI여야만 편한 플레이어

이 게임은 끝까지 터미널 / 스크립트 중심 정체성을 유지한다. 다만 그 위에 읽기 쉬운 workspace를 얹어 진입장벽과 반복 피로를 낮춘다.

## 2. 알파 범위의 핵심 플레이 약속

알파의 목표는 **첫 부팅부터 1차 라이선스 승급까지**를 완성된 세로 슬라이스로 보여주는 것이다.

이 구간에서 플레이어는 최소한 아래를 경험해야 한다.

1. 이 워크스테이션은 **처음에는 터미널밖에 없는 시스템**이라는 사실
2. 기본 명령어로 시스템을 읽고 움직이는 감각
3. `run nexus_shell`을 통해 workspace가 확장되는 순간
4. shell이 플레이를 대체하는 것이 아니라, **터미널 플레이를 보조한다**는 사실
5. 첫 번째 “진짜 침투”에 가까운 계약
6. 손으로 하던 일을 나중엔 스크립트/도구로 줄일 수 있겠다는 예감
7. 1차 라이선스 승급으로 다음 단계가 열린다는 구조

알파는 “데스크톱 기능 과시”가 목표가 아니다. **서버 침투 퍼즐 게임의 핵심 루프가 살아 있는 상태에서, shell이 그것을 얼마나 잘 받쳐주는지**를 보여주는 것이 목표다.

## 3. 첫 시작 연출

### 3.1 부팅 연출
초기 흐름은 기존 방향을 유지한다.

1. **dmesg 스타일 짧은 부팅 로그**
2. **systemd 스타일 서비스 기동 메시지**
3. **getty 로그인 프롬프트**
4. **최초 프롬프트 진입**
5. **자동 타이핑 애니메이션으로 `cat /home/<userId>/README` 유도**

이 구간의 목적은 두 가지다.

- 플레이어에게 이 게임이 “가짜 데스크톱 시뮬레이터”가 아니라 **터미널-우선 워크스테이션**이라는 인상을 주는 것
- 튜토리얼 문구를 길게 띄우지 않고도 세계관과 시스템 분위기를 전달하는 것

### 3.2 getty 로그인 프롬프트
- 최초 1회만 표시한다.
- 입력한 user id는 이후 프롬프트에 반영된다.
- 비밀번호는 요구하지 않는다.

### 3.3 자동 타이핑과 README 유도
최초 프롬프트 진입 직후, 입력을 잠시 막고 다음 명령을 자동 타이핑한다.

```bash
<userId>@nexus:~$ cat /home/<userId>/README
```

플레이어는 엔터만 치면 첫 문서를 보게 된다.

이 연출은 아래를 동시에 수행한다.
- 입력 공포를 낮춘다.
- “터미널은 직접 입력하는 곳”이라는 규칙을 보여준다.
- README가 단순 세계관 문서가 아니라 **첫 행동을 결정하는 운영 문서**라는 인상을 준다.

## 4. Terminal-only 시작 단계

### 4.1 왜 처음부터 shell을 주지 않는가
플레이어는 처음부터 NEXUS Shell로 시작하지 않는다.
처음에는 **터미널만 있는 상태**로 시작해야 한다.

이유는 아래와 같다.

1. 게임의 정체성을 분명히 하기 위해서
   - 이 게임은 터미널 위에 shell이 얹히는 구조지, shell 위에 터미널이 붙는 구조가 아니다.

2. 플레이어가 최소한의 CLI 감각을 몸으로 익히게 하기 위해서
   - `README 보기`
   - `help 보기`
   - `파일 확인 / 메일 확인`
   - `명령을 직접 입력해 결과를 받는 감각`

3. `run nexus_shell`이 단순한 UI 전환이 아니라 **워크스테이션 확장 이벤트**처럼 느껴지게 하기 위해서

### 4.2 terminal-only 단계의 길이
이 단계는 **짧아야 한다.**

목표는 CLI 숙련을 오래 시험하는 것이 아니라, 플레이어에게 다음 세 가지를 체감시키는 것이다.

- “나는 지금 터미널 기반 워크스테이션을 쓰고 있다.”
- “명령을 입력하면 시스템이 반응한다.”
- “조금 더 편한 작업 환경을 여는 방법이 앞으로 등장한다.”

권장 길이는 **첫 5~10분 이내**다.

### 4.3 terminal-only 단계에서 플레이어가 반드시 하게 할 것
README 이후 플레이어는 최소한 아래 정도의 조작을 직접 해봐야 한다.

- 기본 도움말 / 현재 위치 / 홈 디렉토리 확인
- 메시지 또는 메일 확인
- 하나 이상의 텍스트 파일 열람
- 아주 간단한 프로그램 실행 또는 도구 사용 경험

이 구간의 구체 command syntax는 `07`이 소유한다. 본 문서는 **“무엇을 배우게 할 것인가”**만 소유한다.

## 5. README의 역할

README는 단순한 설정 문서가 아니라, **첫 번째 운영 지시서**다.

README는 최소한 아래 역할을 수행해야 한다.

1. 플레이어가 지금 어떤 시스템에 들어왔는지 설명
2. 이 워크스테이션의 이전 사용자에 대한 짧은 암시 제공
3. 다음에 확인해야 할 로컬 정보(메일, 메모, 파일 등)로 연결
4. 플레이어가 적어도 몇 개의 기본 조작을 하게 유도
5. 적절한 시점에 `run nexus_shell`을 실행하도록 연결

### 5.1 README가 당장 하면 안 되는 것
- shell을 맨 처음 줄부터 강요해서 터미널 경험을 건너뛰게 만들면 안 된다.
- 너무 많은 배경설명을 써서 플레이어가 읽기만 하게 만들면 안 된다.
- 저수준 명령어 설명서를 길게 붙이면 안 된다.

README는 **짧고 기능적이어야 하며, 다음 행동을 밀어주는 문서**여야 한다.

## 6. `run nexus_shell` 도입 규칙

### 6.1 대표 진입 문법
온보딩과 스토리 유도에서는 `./nexus_shell`이 아니라 **`run nexus_shell`**을 대표 문법으로 사용한다.

이유는 아래와 같다.

- 플레이어에게 “이건 하나의 프로그램 실행이다”를 더 명확하게 전달한다.
- 초반 연출상 임팩트가 더 크다.
- shell 도입을 단순 경로 실행이 아니라 **의식적인 워크스테이션 확장 행위**처럼 보이게 한다.

직접 실행 문법(`./nexus_shell`)의 지원 여부와 저수준 실행 계약은 `07`, `14`가 소유한다. 본 문서의 관점에서 중요한 것은 **온보딩에서 어떤 문법을 대표적으로 밀 것인가**다.

### 6.2 shell은 언제 도입되는가
shell은 첫 프롬프트 직후 즉시 주어지지 않는다.
플레이어가 최소한의 terminal-only 조작을 해본 뒤, **“이제 작업 환경을 확장하라”**는 흐름으로 도입된다.

권장 구조는 아래와 같다.

1. README 열람
2. 메일/메모/초기 안내 확인
3. 터미널 기본 조작 1~2회 수행
4. `run nexus_shell` 지시 또는 자연스러운 유도
5. shell workspace 첫 진입

### 6.3 shell 첫 실행의 의미
`run nexus_shell`은 단순히 “GUI 열기”가 아니다.
이 순간은 아래 감정을 줘야 한다.

- 나는 여전히 터미널 기반 워크스테이션 안에 있다.
- 하지만 이제 더 강력한 작업 표면을 얻었다.
- 앞으로는 이 workspace를 중심으로 내 도구와 정보 접근성이 확장될 것이다.

즉 shell은 **새 게임을 시작하는 버튼**이 아니라, **같은 게임에서 작업 환경이 한 단계 커지는 순간**이어야 한다.

## 7. NEXUS Shell 첫 진입 경험

### 7.1 첫 진입 시 플레이어가 봐야 하는 것
NEXUS Shell을 처음 열면, 플레이어는 적어도 아래를 immediately 이해할 수 있어야 한다.

- 여전히 터미널이 핵심 작업 공간이다.
- 오른쪽에는 참고 정보나 시스템 앱이 붙는다.
- taskbar/start menu를 통해 pane을 열고 전환할 수 있다.
- 이 shell은 정보를 보여주고 정리해주는 workspace이며, 핵심 입력을 대신하지 않는다.

### 7.2 첫 진입 직후 모든 기능을 설명할 필요는 없다
알파에서는 아래만 우선 명확하면 된다.

- terminal pane
- world/map 성격의 참고 pane 1개
- mail 또는 mission/log 성격의 참고 pane 1개
- taskbar와 start menu의 존재

설정, pin/unpin, maximize, popup 등은 **필요한 순간에 짧게 이해되면 충분하다.**
처음부터 모든 shell 기능을 설명하는 것은 오히려 역효과다.

### 7.3 shell은 플레이를 대체하지 않는다
shell이 열린 뒤에도 아래 원칙은 유지된다.

- 접속 시도
- 스크립트 작성
- 프로그램 실행
- 데이터 조작

같은 실제 게임플레이 입력은 계속 터미널/스크립트 쪽에서 발생한다.

shell은 **정보 표면을 추가하는 것**이지, 명령 입력의 주권을 가져가는 것이 아니다.

## 8. 첫 번째 정식 계약으로의 연결

shell 도입 직후 플레이어는 곧바로 “아무 창이나 만져보는 상태”에 오래 머물면 안 된다.
shell은 **첫 번째 정식 계약 / 침투 과제**로 빨리 연결되어야 한다.

권장 흐름은 아래와 같다.

1. shell 도입
2. 메일/미션 pane 또는 start menu를 통해 첫 계약 노출
3. 계약 목표가 단순하고 명확함
4. 플레이어는 terminal + shell 보조정보를 같이 사용해 해결

즉 shell은 도입 자체가 목적이 아니라, **첫 번째 ‘실전 같은 문제’를 더 잘 읽고 풀게 만드는 준비 단계**여야 한다.

## 9. 1차 라이선스까지의 알파 플레이 흐름

알파 세로 슬라이스는 아래 순서를 권장한다.

### Phase A — Boot / Orientation
- 부팅 연출
- getty 로그인
- README 유도
- terminal-only 상태 정착

### Phase B — Local Literacy
- 기본 파일/메시지 확인
- 최소한의 터미널 조작 경험
- “이 워크스테이션은 직접 만지는 곳”이라는 감각 형성

### Phase C — Shell Unlock
- `run nexus_shell` 유도
- shell 첫 진입
- terminal + pane workspace 구조 인식

### Phase D — First Real Contract
- 첫 번째 정식 계약 수락
- 간단한 정찰 / 접속 / 목표 달성
- shell이 왜 유용한지 체감

### Phase E — Optional Side Work (Low Volume)
- 세계관 보강 / 해커 생활감 / 소액 수익 / 연습용의 주변부 의뢰를 매우 소량 제공
- 반복 파밍 구조가 되지 않도록 주의

### Phase F — First License Promotion
- 지금까지 배운 것들을 한 번 조합하는 승급 과제
- 승급 이후 다음 단계의 기능 확장과 더 복합적인 계약을 예고

## 10. 1차 라이선스 승급 시점의 플레이어 상태

1차 라이선스 승급 직전 또는 직후, 플레이어는 최소한 아래를 알고 있어야 한다.

- 터미널에서 기본 명령과 프로그램 실행을 할 수 있다.
- `run nexus_shell`을 통해 workspace를 꺼내고 다시 활용할 수 있다.
- shell의 pane들은 읽기/참고/전환을 돕는 층이라는 것을 안다.
- 첫 번째 정식 계약을 스스로 추적하고 해결할 수 있다.
- 반복 작업이 생기면, 나중엔 이걸 스크립트/자동화로 줄일 수 있겠다는 감각을 얻었다.

이 시점에서 플레이어는 “나는 스토리를 읽는 사람”이 아니라, **작업 환경을 갖춘 해커가 되어 가는 중**이라는 느낌을 받아야 한다.

## 11. 코딩 필요성 노출

코딩은 튜토리얼 교과서처럼 가르치지 않는다. 대신 아래 방식으로 욕구를 만든다.

1. 처음엔 명령과 공식 프로그램만으로도 풀 수 있게 한다.
2. 하지만 같은 패턴이 반복되면 귀찮아지는 순간을 만든다.
3. shell은 그 반복을 시각적으로 더 잘 드러내 준다.
   - 정보는 계속 보이는데, 조작은 여전히 손으로 한다.
4. 플레이어가 스스로 “이건 스크립트로 묶고 싶다”는 생각을 하게 만든다.

즉 shell은 코딩을 대체하는 UI가 아니라, **코딩 필요성을 더 잘 보이게 만드는 보조 표면**이기도 하다.

---

## 12. 저장 / 로드 / 재시작의 플레이 경험

### 12.1 최초 부팅
- dmesg → systemd → getty → 자동 타이핑 유도까지 전부 보여준다.

### 12.2 reboot
- dmesg / systemd는 다시 보이지만, 최초 1회 전용 getty / 자동 타이핑 연출은 생략 가능하다.

### 12.3 save / load
- save-load는 “새로 부팅한 느낌”이 아니라, **작업 중이던 워크스테이션으로 복귀하는 느낌**이어야 한다.
- 따라서 load 직후는 가급적 빠르게 원래 상태로 복귀해야 한다.

저수준 persistence 경계는 `12`가 소유하지만, 플레이어 경험 관점에서는 아래가 중요하다.

- load는 shell을 다시 배우는 순간이 아니어야 한다.
- 플레이어는 “내가 하던 작업 자리”로 돌아왔다는 느낌을 받아야 한다.

## 13. NEXUS 스토리 훅

- 플레이어는 **NEXUS**라는 이름의 워크스테이션으로 시작한다.
- 이름은 변경할 수 없다.
- 이전 사용자의 흔적이 곳곳에 남아 있다.
- shell 자체도 “원래부터 이 워크스테이션에 있던 작업 환경”처럼 느껴져야 한다.

즉 `run nexus_shell`은 플레이어가 외부 앱을 설치하는 행위가 아니라,
**이 워크스테이션에 이미 존재하던 본격 작업 환경을 처음으로 호출하는 행위**에 가깝다.

이 흐름은 “이전 사용자는 어떤 사람이었고, 이 환경으로 무엇을 했는가?”라는 호기심으로 자연스럽게 이어질 수 있다.

## 14. 구현 우선순위 관점 메모

알파에서 15번 문서를 구현 관점으로 해석하면 우선순위는 아래와 같다.

1. 첫 부팅 연출 유지
2. README 이후 terminal-only 짧은 흐름 확정
3. `run nexus_shell` 유도 문맥 확정
4. shell 첫 진입 후 첫 계약 연결
5. 1차 라이선스 승급 루프 확정

즉 현재 15번 문서에서 가장 먼저 비워져 있던 부분은
**README 이후 어떻게 shell까지 연결되는가**이며,
이번 재정의에서는 그 부분을 우선적으로 메운다.

## 15. 수용 기준 (Alpha Flow)

알파 기준으로 아래가 충족되면 본 문서의 핵심 목표를 달성한 것으로 본다.

1. 플레이어는 처음에 terminal-only 상태로 게임을 시작한다.
2. README와 후속 안내를 통해 최소한의 수동 조작을 수행한다.
3. 플레이어는 자연스럽게 `run nexus_shell`을 입력하게 된다.
4. shell 첫 진입은 UI 전환이 아니라 workspace 확장처럼 느껴진다.
5. shell 도입 직후 플레이어는 빠르게 첫 정식 계약으로 연결된다.
6. 1차 라이선스 승급까지의 흐름에서 shell은 계속 유용하지만, 핵심 입력을 빼앗지 않는다.
7. 플레이어는 이 시점까지 “터미널 기반 해킹 게임을 하고 있다”는 감각을 유지한다.
